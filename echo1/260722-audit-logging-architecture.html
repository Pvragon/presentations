Part II — The Plan

3 The three-layer model

Echo1 answers the requirements with three complementary layers. They overlap by design — a single change is often recorded in more than one. Read the stack top-down (broad infra) to bottom (specific, human).

Layer 3 · InfrastructureDB / API-level, managed

pg_audit (DDL + writes; reads on PHI tables) · RDS audit logs · CloudTrail → CloudWatch → Kinesis Firehose → S3 Object Lock (WORM) → Athena → QuickSight

Answers: "was the database or API itself touched — including PHI reads?"

Layer 2 · Complianceaudit_log · merge_log

Trigger-written, INSERT-only, partitioned, PHI-bearing. The HIPAA / NIST-AU record of truth for data changes.

Answers: "who changed which row, from what value to what, and when?"

Layer 1 · Operationaltransport_history

App/worker-written typed events, append-only, human-readable — a first-class dispatch feature.

Answers: "what happened to this ride, in plain language?"

The two pillars engineers touch daily are Layer 2 audit_log (compliance) and Layer 1 transport_history (operational). They are deliberately separate tables with separate purposes — see §7.

4 Layer 3 — Infrastructure audit

Provided largely by managed AWS + PostgreSQL primitives; little application code. This is the layer that satisfies "log PHI access," not just PHI changes.

  • pg_audit — DDL + all write operations on every table, plus read operations on PHI tables (client, transport, claim).
  • RDS/Aurora audit logs → published to CloudWatch Logs.
  • CloudTrail — forensic trail of all AWS API activity, with log-file integrity validation.
  • Log pipeline — Fargate audit middleware → CloudWatch (KMS-encrypted) → Kinesis Firehose → S3 Object Lock (6-year+) → Athena → QuickSight.

Satisfies primarily REQ-01 (access side), REQ-02, REQ-03, REQ-05.

5 Layer 2 — Compliance trail (audit_log)

The blanket, polymorphic change log — the HIPAA/NIST-AU record of truth for data changes.

  • Trigger-written, in-transaction. A PostgreSQL trigger writes to audit_log on every INSERT/UPDATE/DELETE of an audited entity, in the same transaction. Application code cannot forget it.
  • Polymorphic. One table covers ~30+ entities via entity_type + entity_id.
  • INSERT-only, forever. No UPDATE/DELETE grants at any privilege level. A dedicated audit_writer role (trigger-only, SECURITY DEFINER) is the sole writer.
  • Scoped, not blanket. Each entity carries an explicit Audit: Yes/No. Exempt: (1) tables that are audit trails, (2) cache/UI state, (3) operational tooling & reconstructible aggregates.
  • Partitioned monthly on created_at, with a DEFAULT partition + PagerDuty alert as a safety net.
  • Retention — seal at month+7, 37 mo hot, 49 mo cold (S3 Object Lock, compliance mode) = 7 years.
  • Read access — RLS restricts to the tenant; tRPC further gates to tenant_admin / platform_admin.

Performance mitigations — the V1 lesson (REQ-12)

  1. Changed-columns-only payloads (Sprint 1). The trigger diffs OLD vs NEW and writes only changed columns to old_data/new_data (mirrored by changed_columns TEXT[]), not full-row snapshots — ~90%+ smaller rows for typical single-field updates.
  2. Samsara telemetry exemption (Sprint 1). If only telemetry columns changed on stop (actual_arrival_at, actual_departure_at, odometer, en_route_at, stop_status, samsara_stop_id), skip the INSERT — the data already lives on stop and in transport_history, so no audit gap. Human edits still audit.
  3. Async escape hatch (evaluate at 10+ tenants). If 1+2 are insufficient, replace the synchronous INSERT with pg_notify + a dedicated Graphile Worker listener. Only if trigger latency exceeds 5ms p95.

Controlled PHI purge · REQ-09

A separate audit_purger role holds UPDATE (old_data, new_data) only — it can redact PHI within the JSONB payloads (human-approval-gated, externally logged) while the who / when / what-entity / what-action structure is preserved and immutability holds for every other role.

Sibling immutable log · merge_log

Same INSERT-only controls as audit_log, dedicated to client/facility de-duplication merges. Stores full pre-merge snapshots of everything merged away (keyed by entity UUID), FK-repointing logs, and outcome — so a merge is fully reconstructible even after source rows are soft-deleted.

6 Layer 1 — the ride timeline (transport_history)

This is the table you were picturing — the append-only ride lifecycle event log (transport_events / transport_history). It is the heart of the operational half of the audit story, and the one place where "what happened to this ride" is told as a human-readable sequence. Its job is REQ-10 — a first-class dispatch feature, not a compliance artifact.

One log per ride

Keyed on transport_request_id, not transport_id. A ride is one logical entity that moves through stages — request → dispatch → execution → completion — and since transport_request → transport is 1:1, the whole chronological story lives in a single keyed log. Pre-dispatch events (confirmations, void) and post-dispatch events (status changes, trip-log steps, reassignments, cancellation) share one timeline. Open a ride, read top to bottom, and you have its life.

Why it exists separately from audit_log

The single most important thing to understand about the audit design: transport_history and audit_log are not redundant. They record overlapping moments for opposite reasons, aimed at opposite audiences. Collapsing them into one table would break both jobs at once.

If you tried to serve the timeline from audit_log

You would expose raw, PHI-laden column diffs to dispatchers; force them to reverse-engineer meaning from column names; be unable to represent events that aren't a single-row column change (a trip-log step, an escalation, a duplicate-lineage marker); and couple a user-facing feature to the immutable compliance store. The separation is what lets each store be excellent at its one job.

The rule that makes the catalog binding

The event list is not advisory. Per the coverage rule (§7): any flow that mutates transport lifecycle state must emit the matching event here, in the same operation. A flow that changes a ride's state without writing transport_history is a spec violation — the timeline is only trustworthy if it is complete.

Enforcement, retention & access

  • Append-only, three enforcement layers (all required for CISO sign-off): (1) tRPC exposes only an insert procedure; (2) history_writer role has INSERT+SELECT only; (3) a BEFORE UPDATE OR DELETE trigger raises an exception. See Appendix B.
  • Partitioned monthly on occurred_at; 36-month hot retention; detach → archive to S3 → drop (no DELETE needed — archival respects append-only).
  • Role-gated reads — RLS for tenant isolation; drivers see only events for their assigned transports; dispatchers and managers see all tenant events.
  • PHI-aware — GPS coordinates and free-text reasons live in detail; sub-fields registered in the PHI Field Registry (ENC-04) with Sentry scrubbing and log exclusion.

7 The coverage rule — how the layers stay in sync

Load-bearing invariant

Any UPDATE to a transport row (or related lifecycle-bearing row: transport_request, route, stop, billing fields) by an application flow MUST emit a corresponding transport_history event in the same logical operation.

So a single state-changing flow produces both records: the transport_history event (operational) and the trigger-fired audit_log row (compliance). A flow that mutates transport state without writing transport_history violates the rule. "audit_log is the compliance trail; transport_history is the operational timeline; both must coexist for state-changing flows."

Carve-out: external-system sync-target columns (route.samsara_route_id, route.route_polyline, route.external_ids, stop.samsara_stop_id, owned by the Samsara Outbound queue) are exempt — they capture cross-system mirror state, a separate concern with its own observability, not human-actor lifecycle actions.

8 Attribution integrity (REQ-11)

Both pillars carry a nullable actor_user_id (NULL = system/worker/trigger-initiated). Correct attribution over time depends on a hard invariant: users are never hard-deleted (is_active = false + deactivated_at). If hard-deletion of users is ever proposed, this dependency must be re-evaluated. audit_log additionally always carries trigger_source, so every record has at least one attribution vector even when the actor is NULL.

10 Known open items & caveats

  • PG version. On PG 15, FK constraints on partitioned tables (audit_log, transport_history, merge_log, db_health_finding) are application-layer enforced with CI tests; intent to restore DB-enforced FKs on PG 17+.
  • WORM backup deadline. audit_log → S3 Object Lock export was earmarked pre-production (target 2026-07-01, owner Farhan). Confirm status.
  • GIN on detail. Deferred on both JSONB payload columns (write amplification on high-write append-only tables). Revisit post-launch with CREATE INDEX CONCURRENTLY if query patterns demand it.
  • AU-6 automated review. Anomaly detection is currently "Planned" — surfacing PHI-access anomalies from pg_audit/audit_log is not yet built.

Immutability & roles

CREATE TYPE audit_action_enum AS ENUM ('INSERT', 'UPDATE', 'DELETE');

CREATE ROLE audit_writer NOLOGIN;
GRANT INSERT ON audit_log TO audit_writer;
REVOKE ALL ON audit_log FROM PUBLIC;      -- no UPDATE/DELETE/TRUNCATE. Ever.

-- Scoped PHI-redaction role (human-approval-gated, externally logged)
CREATE ROLE audit_purger NOLOGIN;
GRANT UPDATE (old_data, new_data) ON audit_log TO audit_purger;

CREATE FUNCTION audit_trigger_fn() RETURNS TRIGGER
  SECURITY DEFINER SET ROLE = 'audit_writer' ... ;

RLS: FORCE ROW LEVEL SECURITY; SELECT tenant_id = current_setting('app.tenant_id'), further gated to admin at tRPC. Partition: PARTITION BY RANGE (created_at) + DEFAULT partition (PagerDuty alert).

Append-only enforcement — three layers

CREATE ROLE history_writer;
GRANT INSERT, SELECT ON transport_history TO history_writer;   -- no UPDATE/DELETE

CREATE OR REPLACE FUNCTION prevent_transport_history_mutation()
RETURNS TRIGGER AS $$
BEGIN
  RAISE EXCEPTION 'transport_history is append-only: % not permitted', TG_OP;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER transport_history_no_mutation
  BEFORE UPDATE OR DELETE ON transport_history
  FOR EACH ROW EXECUTE FUNCTION prevent_transport_history_mutation();

Indexes: ride timeline (tenant_id, transport_request_id, occurred_at), event-type (tenant_id, event_type, occurred_at), actor partial … WHERE actor_user_id IS NOT NULL. Partition: PARTITION BY RANGE (occurred_at), monthly, 36mo hot. Custom autovacuum (DO-HIGH-2).
